Debounce the notification worker to stop duplicate emails

fix/notify-debounce → main @yourhandle 7 files +184 −63

TL;DR — Users received the same notification email up to five times during traffic spikes because the worker fired once per event with no coalescing. This PR adds a 30-second debounce keyed on (user_id, event_type), backed by Redis, and drops duplicates before send. No schema changes; safe to roll back by toggling the NOTIFY_DEBOUNCE flag.

Why this change

The motivation, the impact, and what triggered the work.

Support flagged 40+ tickets last week about repeated "Your order shipped" emails. Root cause: each shipment event re-enqueued a job, and the worker had no idea a near-identical job had just run. During the Black Friday rehearsal this multiplied — one user got eleven copies.

Before

  • Every event triggered an immediate send, no coalescing.
  • Duplicate jobs ran in parallel with no shared state.
  • 5×+ duplicate emails under load; angry users.

After

  • Events are debounced 30s per (user, type).
  • A Redis lock guarantees one send per window.
  • Zero duplicates across a 50k-event load test.

The heart of the change. Before sending, the worker now claims a debounce key. If the key already exists, the job exits early — the in-flight job will send the latest state.

Rollout & rollback

Ship plan and the escape hatch.

Guarded by the NOTIFY_DEBOUNCE flag, default off. Plan: enable for internal users → 5% → 100% over three days, watching the duplicate-send dashboard. Rollback is a single flag flip; no data migration to reverse.
