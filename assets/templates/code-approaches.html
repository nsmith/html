Three ways to implement debounced search for the task filter input

We type-filter a long task list on every keystroke. Below are three implementation paths, each with its trade-offs, so we can pick the one that fits our React codebase and bundle budget.

01 Inline useEffect 02 Custom hook 03 External library ★ Recommendation
01

Inline useEffect + setTimeout

Debounce directly inside the component with a timer cleared on each render.

const [draft, setDraft] = useState("");

useEffect(() => {
  const id = setTimeout(() => onSearch(draft), 250);
  return () => clearTimeout(id); // cancel on next keystroke
}, [draft]);

Pros

  • Zero dependencies, nothing new to learn.
  • Fully visible logic in one place.

Cons

  • Copy-pasted everywhere it is needed.
  • Harder to unit-test in isolation.
Bundle+0 kb
TestabilityLow
ReuseLow
SSR safeYes
02

Custom useDebounce hook

Extract the timer logic into a reusable, testable hook.

function useDebounce(value, delay = 250) {
  const [v, setV] = useState(value);
  useEffect(() => {
    const id = setTimeout(() => setV(value), delay);
    return () => clearTimeout(id);
  }, [value, delay]);
  return v;
}

Pros

  • Reusable across every input.
  • Trivial to unit-test with fake timers.
  • Matches our existing hooks folder pattern.

Cons

  • One more file to maintain ourselves.
Bundle+0 kb
TestabilityHigh
ReuseHigh
SSR safeYes
03

Tiny external library

Pull in a maintained debounce utility and wire it to the input.

import { useDebouncedValue } from "some-hooks";

const [draft, setDraft] = useState("");
const query = useDebouncedValue(draft, 250);
useEffect(() => onSearch(query), [query]);

Pros

  • Battle-tested, edge cases handled.
  • No code to maintain ourselves.

Cons

  • Adds a dependency for ~6 lines of code.
  • Extra supply-chain surface to audit.
Bundle+1.4 kb
TestabilityMedium
ReuseHigh
SSR safeYes
Recommendation

Go with approach 02 — the custom hook

It gives us reuse and testability with zero bundle cost, and it slots straight into our existing hooks/ folder so reviewers will recognise the pattern.

Reach for approach 03 only if we later need richer behaviour (leading-edge, max-wait) that would otherwise reimplement a known library. Approach 01 is fine for a one-off prototype but should not ship to shared code.
